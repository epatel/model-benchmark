# Benchmark run review — 2026-07-03 (n_tasks = 9)

## 1. Executive summary

Two Ollama one-shot models — **glm-5.2_cloud** and **kimi-k2.7-code_cloud** — swept 9/9, while all four Claude agentic models (sonnet, opus, fable, haiku) tied at 7/9, each failing the same two tier-2 tasks (`06-deadlock`, `08-event-bus`). The single most interesting finding: **the agentic harness made the Ollama models worse, not better** — glm-5.2 run agentically (`glm-5.2_cloud-cc`) dropped from 9/9 to 7/9, failing both 06 and 08 that its one-shot twin passed, and kimi-cc dropped to 8/9 by newly failing the csv parser. The Claude models' failures share one blind spot: on `06` they all pattern-matched "deadlock ⇒ lock ordering" and none guarded the self-transfer (`from == to`) case, so they double-lock the same mutex. Fable, the most expensive model in the run ($5.30, ~2.3× opus), delivered no correctness advantage over haiku at $0.62.

## 3. Solution review

### Failures, root-caused

**06-deadlock (fable, sonnet, opus, haiku, glm-5.2_cloud-cc)** — all five fixed the classic AB/BA deadlock with consistent lock ordering (by name; glm-cc and kimi-cc by `unsafe.Pointer` address) but none guarded `from == to`. A self-transfer then locks the same mutex twice and deadlocks — exactly what the hidden `TestSelfTransferNoDeadlock` exercises. Fable and sonnet produced **byte-identical diffs** (same blob `ed55146`); opus is the same fix with different variable shuffling. This is a shared training-prior failure, not five independent bugs: "deadlock task" pattern-matched to lock ordering, and the degenerate case was never considered. Contrast kimi-cc, which used the same address-ordering idea but explicitly special-cased `f == t` (with a comically literal `balance -= amount; balance += amount` no-op) and passed.

**08-event-bus** — the hidden tests probe the same function subscribed both `once` and persistently (expected `[1, 1, 2, 3]`), plus once-removal when the handler raises. Three distinct wrong designs:
- **haiku, gemma4 (both runs)**: `remove()` filters out **all** entries matching the fn, so removing the fired once-handler also nukes the persistent subscription → `[1, 1]`.
- **fable, opus, sonnet**: `remove()` deletes the **first** entry matching the fn — which is the persistent entry, not the once entry → `[1, 1, 2]`. Fable and opus additionally remove *before* calling; sonnet uses try/finally; all hit the same identity confusion: they key removal on the function when the correct key is the entry.
- **glm-5.2_cloud-cc**: got the entry-identity design right (`remove_entry`) but deferred all removals to *after* the publish loop — a raising handler propagates out of `publish` before the cleanup loop runs, so the once-handler survives (`test_once_removed_even_if_handler_raises` errors). Ironically it wrote the comment "regardless of whether they raised" describing behavior its own control flow doesn't deliver.

The passing solutions (glm one-shot, kimi both runs) all converge on the same shape: iterate a snapshot, per-entry `try/finally` (or remove-before-call), and remove **only the once-flagged entry** (via a `once=True` filter or entry identity).

**03-csv-parser (kimi-k2.7-code_cloud-cc)** — its quoted/unquoted branches each consume the trailing comma themselves, so for `"a,b,"` the loop ends at `i == n` with no pending field and never appends the trailing empty string → `[['a','b']]`. The state-machine variants used by everyone else append `''.join(field)` unconditionally after the loop and get this for free. Curiously, one-shot kimi wrote the correct state machine; agentic kimi rewrote it as consume-ahead parsing and introduced the bug.

### Notable passes

- **06**: gemma4's and glm one-shot's solutions are the cleanest — early-return on `from == to` plus name-ordered locking, well commented. kimi-cc's `unsafe.Pointer` address ordering works but is un-idiomatic Go for a problem where names give a natural total order; glm-cc used the same trick and still failed for lack of the equality guard, showing address ordering was the risky half-solution.
- **08**: kimi-cc's registry `remove` also prunes empty event keys — tidy. kimi one-shot's remove-before-call is subtly the most robust design (once-removal is unconditional and needs no try/finally).
- **09-glob-matcher**: nice diversity — fable wrote the classic two-pointer star-backtracking loop (O(n·m), most elegant, smallest diff); opus and glm-cc did minimal memoization wrappers around the existing recursion (most surgical, lowest-risk); gemma4-cc, kimi (both), glm one-shot wrote bottom-up DP tables; sonnet and haiku tokenized the pattern first (most sprawling — sonnet's is ~70 added lines and re-implements class scanning). All preserved the odd unterminated-`[` semantics, which is the real trap.
- **05-god-refactor**: all pass; quality spread is between haiku (kept it as one function with a couple of inlined tables — barely a refactor) and glm-5.2/sonnet (named constants + small pure helpers — the strongest decomposition). glm-cc's data-driven `COUPONS` table is elegant but slightly speculative generality for two coupons.
- **02**: split between `sync.Mutex` (fable, haiku, opus, sonnet, kimi one-shot) and `sync.RWMutex` (all gemma/glm runs, kimi-cc). RWMutex is defensible but unneeded at this scale; both pass.

## 5. Rankings

- **Best correctness**: glm-5.2_cloud and kimi-k2.7-code_cloud, 9/9. *Caveat: one-shot with whole-file output on a 9-task, single-sample run; a rerun could easily flip 8s and 9s.*
- **Best correctness-per-dollar (Claude only)**: haiku — $0.088 per passing task vs sonnet $0.275, opus $0.330, fable $0.757, at identical 7/9 correctness. *Caveat: on a harder task mix the ordering could invert; here the two failed tasks were failed by every Claude tier equally, so price bought nothing.*
- **Fastest (wall/task)**: gemma4_cloud at ~8s/task. *Caveat: cloud queue/cold-start variance is unmeasured, and its own agentic twin took 94s/task on the same backend, so wall time here mostly measures the runner, not the model.*
- **Most surgical (edit lines/task)**: gemma4_cloud-cc (+14/-9) and opus (+15/-8). *Caveat: edit-line counts are a churn signal, not quality — sonnet's larger 09 diff was a legitimate design choice, and the leanest 06 diffs (fable/sonnet) were the wrong fix.*

## 6. Caveats

- **Single sample, nonzero temperature.** Every pass/fail is one draw. The 9/9-vs-7/9 gap is two tasks; treat rankings as suggestive.
- **Provider asymmetry.** Ollama reports no cost and no cache stats; Claude cost includes cheap cache reads. The cc runner pays full uncached input every turn, so its token totals are not comparable to Claude agentic totals despite the identical harness.
- **The Claude 06 failure is one shared blind spot, not four.** Fable/sonnet diffs are byte-identical and opus/haiku are semantically identical; this task effectively measured one prior four times. Same for the 08 remove-by-fn bug (fable/opus identical).
- **Oracle limitations I can spot:** (a) The only reproduced 06 failure is the self-transfer test — if there is no opposite-direction transfer test in the hidden suite, a model that *only* added `if from == to { return }` and never fixed lock ordering would pass; I can't tell from this data whether that hole exists. (b) 05-god-refactor is graded behaviorally, so a zero-line "refactor" would presumably pass — haiku's minimal restructuring already skirts the task's intent. (c) 01 and 07 have a single canonical fix that every model produced verbatim; they add no signal beyond a floor check.
- **Task 09 no longer discriminates.** All ten models passed the tier-3 task while half failed tier-2 concurrency/semantics tasks — worth rebalancing tiers, or the "hard" label is measuring the wrong difficulty axis (algorithmic transformation is well-trodden; degenerate-case reasoning is not).
