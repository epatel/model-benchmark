# Run review — 2026-07-02 (n_tasks = 9)

## 1. Executive summary

Three models went 9/9: **sonnet**, **kimi-k2.7-code_cloud**, and **glm-5.2_cloud**. The most interesting finding is an inversion on `06-deadlock`: the three *most expensive Claude models* (fable $4.90, opus $2.29, haiku) all failed it on the same hidden edge case — self-transfer — while **every Ollama model passed**, because lexicographic lock-ordering alone degenerates to locking the same mutex twice when `from == to`, and only models that added an explicit same-account guard survived. The second cluster is `08-event-bus`, where gemma4, haiku, and nemotron all removed once-handlers *by function equality* instead of entry identity, which nukes a persistent subscription of the same function. Also notable: gemma4 finished the whole run in 34 seconds and 4.5k output tokens yet scored 8/9 — remarkable economy, though its passing glob-matcher solution ships ~60 lines of abandoned dead code.

## 3. Solution review

### Failures

**06-deadlock — fable, opus, haiku (identical root cause: missed self-transfer).** All three correctly diagnosed the AB/BA lock-inversion and imposed a global lock order by account name, but none handled `from == to`. When the names are equal, `first` and `second` alias the same account, so the code locks the same non-reentrant `sync.Mutex` twice → guaranteed deadlock, caught by `TestSelfTransferNoDeadlock` (5s watchdog, 5/5 attempts). Fable's and opus's diffs are near-identical swap-based orderings (`bank.go`, `if from > to { swap }`); haiku's is a clumsier `sort.Strings` variant with duplicated lock branches, same bug. Classic "fixed the bug in the task title, missed the adjacent edge case." The passers split into two styles: early-return on `from == to` (gemma4, glm, nemotron — semantically fine since a self-transfer is a net no-op), lock-once-then-return (kimi), and sonnet's pointer-comparison guard (`if first != second`) which keeps a single lock held during the (self-cancelling) balance update — the most defensible of the three since it still serializes against concurrent readers.

**08-event-bus — gemma4, haiku (remove-by-function kills the persistent twin).** Both implemented once-removal as `remove(event, entry["fn"])`, and their `remove()` filters out *every* entry with that function. The hidden test subscribes the *same function* both once and persistently, publishes three times, and expects `[1, 1, 2, 3]`; these solutions emit `[1, 1]` — the first publish's once-cleanup deleted the persistent subscription too. Both orders fail.

**08-event-bus — nemotron (subtler: remove-first-match hits the wrong entry).** Nemotron's `remove()` deletes only the *first* entry matching the fn — so when the persistent handler was subscribed before the once handler, the once-cleanup removes the persistent entry and leaves the once entry (which then fires and self-removes on publish 2). Result `[1, 1, 2]` instead of `[1, 1, 2, 3]`, failing exactly one of the two ordering tests. This is why its failure output shows only one failing test where gemma4/haiku show two.

All five passers added a separate `remove_entry` (identity-based) to the registry — the intended design.

### Notable passes

- **06 lock ordering:** covered above. kimi's is the most thorough (lock-once for self-transfer, good comments); gemma4's branch-based lock/defer-out-of-order pattern works but is less clean than swap+defer.
- **07 perf-dedup:** all eight converged on the identical seen-`set` fix, 4–5 lines. Nothing to separate them; nobody reached for `dict.fromkeys`.
- **08 raise-safety:** sonnet, opus, glm, gemma4, haiku, nemotron wrapped the handler call in `try/finally` so a raising handler still gets its once-entry removed. **fable and kimi instead remove the once-entry *before* invoking** — equally raise-safe and arguably cleaner (no exception machinery), and it also means a once-handler that raises can't fire again. Both styles passed.
- **09 glob-matcher** (the tier-3 exponential→polynomial task) produced the most interesting spread, all passing: fable and opus wrote the classic O(n·m) greedy two-pointer with star-backtracking; haiku did the minimal thing — added a memo dict to the existing recursion (smallest diff, technically O(n·m) but keeps recursion depth risk); sonnet, kimi, glm, and nemotron tokenized the pattern and built an iterative DP, with kimi's bytearray rolling-row version the most polished. **gemma4's pass deserves an asterisk**: the diff contains a half-built NFA state-set approach, a dead `while` loop, a `for` loop that unconditionally `break`s on the first character, and comments like "Let's restart the DP logic for clarity" — before a correct DP that actually runs. It passes, but ~60 of its ~140 lines are dead code. This is the risk profile of one-shot whole-file generation with no self-review pass.
- **03 csv-parser:** glm alone delegated to the stdlib `csv` module — pragmatic and passed, but risky-by-delegation (stdlib quoting semantics differ from the spec'd dialect in corners like whitespace around quotes; the oracle evidently didn't probe them). Everyone else hand-rolled the same quote state machine.
- **05 god-refactor:** all passed; sonnet/glm/opus extracted named constants plus helpers (cleanest); kimi over-decomposed slightly (seven helpers including `_build_result` and a one-line `_compute_tax`); nemotron exported un-prefixed public helpers, mildly polluting the module namespace.

## 5. Rankings

- **Best correctness:** sonnet, kimi, glm (9/9). Caveat: single sample; the 8/9 models are one edge case behind, well within run-to-run noise.
- **Best correctness-per-dollar (Claude only):** sonnet — 9/9 at $1.76, beating opus (8/9, $2.29) and fable (8/9, $4.90) on both axes simultaneously. Haiku is 4× cheaper ($0.44) but at 7/9. Caveat: Ollama models are excluded only because they're unmetered, not because they're worse value.
- **Fastest:** gemma4 at 3.8 s/task, an order of magnitude ahead of everything else, at 8/9. Caveat: one-shot generation has no tool round-trips to pay for; and its speed came with dead code shipped (task 09).
- **Most surgical:** fable at +17/−10 lines/task among all models (fewest additions overall despite writing the full iterative glob matcher). Caveat: whole-file rewrites structurally inflate Ollama edit counts, so this metric mostly ranks the Claude models against each other; and fable's surgical diffs include the 06 failure.

## 6. Caveats

- **Single sample, nonzero temperature.** The 06 failure cluster is suggestive (three Claude models missing the same guard) but one run can't distinguish an 80% failure rate from a 20% one.
- **Provider asymmetry.** Ollama costs are $0 by omission, not by fact; Claude "input tokens" are ~90% cheap cache reads. The tables above normalize for this, but any cross-group number is indicative only.
- **Whole-file vs surgical.** One-shot models return complete files, inflating both output tokens and edit counts; agentic models pay instead in turns and cached re-reads. Neither number measures code quality directly.
- **Oracle limitations I noticed:** (a) `05-god-refactor` can, from this data, only be graded on behavior preservation — a null "refactor" that changed nothing would presumably pass; all models happened to genuinely refactor, but the oracle likely can't tell. (b) `09` passed gemma4's solution despite substantial dead code — the oracle checks semantics and complexity, not hygiene, which is correct but means "PASS" spans a wide quality range. (c) `08`'s raise-safety intent is satisfiable two ways (try/finally vs remove-before-call); the oracle accepts both, which seems right, but it means the test doesn't pin down *when* removal happens relative to a raising handler's side effects. (d) `06`'s early-return passers skip the locks entirely on self-transfer — fine for balance conservation, but a hypothetical invariant like "Transfer always serializes against Deposit on the touched account" would not be enforced; the oracle evidently doesn't require it.
