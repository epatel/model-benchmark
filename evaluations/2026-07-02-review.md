# Run review — 2026-07-02 (9 tasks × 8 models)

## 1. Executive summary

The two 9/9 finishers are both one-shot Ollama models: **glm-5.2_cloud** (338s) and **nemotron-3-ultra_cloud** (1265s). The single most interesting finding is that **all four Claude models failed 06-deadlock the same way**: every one correctly diagnosed the classic AB/BA lock-ordering deadlock and fixed it, but none guarded the self-transfer case (`from == to`), so ordering the two lock acquisitions still double-locks one non-reentrant mutex — while **all four Ollama models added the `from == to` guard** and passed. The other failure clusters are also semantic edge cases, not compile errors: trailing empty field in CSV (`"a,b,"` — haiku, kimi) and the hardened event-bus oracle where the same function is subscribed both once and persistently (gemma4, sonnet, haiku — all three unsubscribe by function identity instead of by entry identity). Fable and Opus tie at 8/9 with their only loss being that shared self-transfer blind spot.

## 3. Solution review

### Failures, root-caused

**06-deadlock — fable, opus, haiku (lock ordering without self-transfer guard).** All three pick `first, second` by account-name order and lock both. When `from == to`, `first` and `second` are the *same* `*account`, so `second.mu.Lock()` blocks forever on the mutex `first.mu.Lock()` already holds. The hidden test `TestSelfTransferNoDeadlock` hits exactly this. The diffs are otherwise correct fixes for the opposing-transfer deadlock — this is "fixed the deadlock they imagined, not the one in the test."

**06-deadlock — sonnet (same bug, riskier mechanism).** Sonnet ordered locks by `uintptr(unsafe.Pointer(...))`. Address ordering is a legitimate technique, but (a) it imports `unsafe` for a problem solvable with the account name already in hand, and (b) it has the identical hole: when `f == t` the addresses are equal, no swap happens, and it double-locks. Same failure, plus an `unsafe` dependency a reviewer would reject.

By contrast, every passer special-cased self-transfer: gemma4 and nemotron early-return on `from == to` (fine — net effect is zero), kimi compares the resolved pointers `f == t` and locks once (the most robust check — catches aliased names), glm locks once and performs a pointless `-= / +=` pair (harmless but odd). Kimi also added an `id` field for total ordering — more invasive than needed when names order fine, but sound.

**08-event-bus — gemma4 (`[1,1]` vs `[1,1,2]`).** Its `registry.remove()` deletes *all* entries whose `fn` matches, and `publish` removes once-handlers via `unsubscribe(event, entry["fn"])`. With the same function subscribed both once and persistently, firing the once-handler nukes the persistent subscription too — the persistent handler never fires again.

**08-event-bus — sonnet and haiku (`[1,1,2]` vs `[1,1,2,3]`).** Mirror image: their `remove()` deletes the *first* entry whose `fn` matches. After the once-handler fires, the removal hits the persistent entry (subscribed first) instead of the once entry; the leftover once entry fires on publish 2 and is then removed, so publish 3 calls nothing. Root cause in both cases: removing by function identity when the removal target is a specific *entry*. Passers (fable, glm, opus, nemotron) added an identity-based `remove_entry(event, entry)`; kimi added `remove_once(event)`. This is exactly what the hardened oracle (per the recent commit) was built to catch.

**03-csv-parser — haiku and kimi (trailing empty field).** Structurally the same bug in both: their loop appends a field, then consumes the separator comma, and exits when `i == n` — so `"a,b,"` yields `["a","b"]` with no trailing `""`. Fable, opus, and sonnet avoid it by construction: a running `field` buffer with one unconditional `fields.append` after the loop, so the trailing comma naturally leaves an empty field. Nemotron has the same loop shape as the failers but patched it with an explicit `if line.endswith(','): fields.append('')` — it passes, but it's a bolted-on heuristic (a line ending in a comma *inside* an unterminated quote would get a spurious extra field). Risky-but-passing.

### Notable passes

- **08 raise-safety and snapshot:** kimi's is the cleanest design — snapshot the handler list, strip once-entries *before* invoking, so once-handlers fire exactly once even if one raises, and mid-publish subscriptions aren't dropped. Fable achieves the same raise-safety by removing the once entry just before calling it (no try/finally needed). glm/opus/nemotron use `try/finally` around each call — also correct. Nemotron reaches into `registry._remove_entry` (private-by-convention method it added itself — cosmetic).
- **09-glob-matcher (tier-3):** four distinct correct strategies. Haiku and sonnet memoized the existing recursion — the most surgical change and the safest way to preserve exact semantics, though Python recursion depth is a latent risk on very long inputs. Fable and opus rewrote to the greedy two-pointer star-backtracking scan (O(n·m), iterative, idiomatic); opus's version calls `_single_match` twice per character (minor waste). glm, nemotron, kimi, and gemma4 wrote tokenizing DPs — correct, and they carefully preserved the unterminated-`[` -matches-literal-`[` quirk, but kimi's ~100-line version with `bytearray` rows is heavily over-engineered for the task.
- **07-perf-dedup:** all eight produced the identical seen-set fix. Nobody used `dict.fromkeys` — no differentiation here.
- **05-god-refactor:** all passed; quality varies. Fable/opus/glm/sonnet extracted named helpers plus constants — clean. Haiku's `COUPON_CONFIG = {"SAVE5": (-5, False)}` encoding coupons as signed deltas is clever-but-obscure (a `max(amount + delta, 0)` applied to FREESHIP's 0 delta happens to be safe, but the abstraction is wrong the moment a coupon isn't a flat discount). **gemma4's 03 diff deserves a flag: it defines `parse_csv` twice**, leaving its first attempt as dead code that the second definition shadows, with meandering comments — it passes only because Python takes the last definition. That's one-shot whole-file output with no self-review.
- **02-bank-ledger:** gemma4 and nemotron used `RWMutex` with read locks on `Balance`/`Total` — marginally better than the plain `Mutex` everyone else used; all correct.

## 6. Caveats

- **Single sample at nonzero temperature.** One run per model; the 8/9 cluster is separated from 9/9 by a single edge case each. Rankings within ±1 task are noise. That said, the 4-for-4 Claude failure on self-transfer looks systematic, not random — same conceptual gap, four independently-sampled models.
- **Provider reporting asymmetry.** Ollama's $0.00 cost and 1-turn structure make cost and turn comparisons meaningless across groups; everything in §4 outside the Claude cost column is indicative only.
- **Oracle discrimination is narrow on 06.** Every model fixed the AB/BA deadlock; the entire pass/fail split comes from `TestSelfTransferNoDeadlock`. A model that only guarded self-transfer but kept broken lock ordering would presumably fail other hidden tests — I can't verify that from this data.
- **05-god-refactor is behavior-graded.** All diffs preserve exact semantics, but a null refactor (or a trivial one) would presumably also pass a behavioral oracle — I can't tell from this data whether structural quality is checked at all, so treat 05's universal PASS as "didn't break anything" rather than "refactored well."
- **Fable's 08 pass hides a latent bug:** its `registry.remove` (used by `unsubscribe`) removes only the first entry matching by `==` — the same by-function removal shape that sank sonnet/haiku — it passes because its *once*-cleanup path uses identity-based `remove_entry`. An oracle test unsubscribing a duplicate-subscribed handler could expose it.
- **Nemotron's 03 pass rests on the `endswith(',')` patch** rather than a structurally correct loop — a quoted-field edge case (trailing comma inside an unterminated quote) would produce a spurious field. Passing, but fragile.
