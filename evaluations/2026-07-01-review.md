# Run review — 2026-07-01 (8 models × 8 tasks)

## 1. Executive summary

Six of eight models went 8/8: kimi-k2.7-code, glm-5.2, nemotron-3-ultra (Ollama one-shot) and sonnet, opus, fable (Claude agentic). gemma4 dropped only 08-event-bus (7/8); haiku dropped 06-deadlock and 08-event-bus (6/8). The most interesting finding: **both failures trace to the same class of bug — removing/handling by *function identity* instead of *entry identity*** — and both failing tests are exactly the oracle hardenings added in commit 7988df7 (06 self-transfer, 08 same-fn once+persistent). Before that hardening, this run would likely have been a clean 8×8/8 sweep — the benchmark's discriminating power currently lives almost entirely in those two edge cases. Also notable: haiku emitted the most output tokens of any Claude model (19.3k) yet scored worst, while gemma4 finished the whole run in 53s with 2.8k output tokens and only missed one edge case.

## 3. Solution review

### Failures

**haiku — 06-deadlock (deadlocked, watchdog).** Haiku correctly added lexicographic lock ordering, but missed the self-transfer case. When `from == to`, `from < to` is false, so `first, second = t, f` — both point at the *same* account, and `first.mu.Lock()` followed by `second.mu.Lock()` locks the same non-reentrant mutex twice → guaranteed deadlock, which is precisely what `TestSelfTransferNoDeadlock` timed out on (5s × 5 reps). Classic "fixed the named bug, not the invariant." All seven passers handled `from == to` explicitly — either early-return no-op (gemma4, glm, kimi, nemotron) or lock-once-then-do-net-zero-arithmetic (fable, opus, sonnet).

**haiku — 08-event-bus (wrong edge case: remove-by-function).** The hidden test subscribes the *same function* both `once` and persistently, expecting `[1, 1, 2]` across two publishes. Haiku's once-cleanup calls `self._registry.remove(event, entry["fn"])`, and its `remove` filters out *every* entry whose `fn` is the handler — so firing the once-entry also deleted the persistent subscription, yielding `[1, 1]`.

**gemma4 — 08-event-bus (same root cause).** Identical failure mode via a different path: after a once-handler fires it calls `self.unsubscribe(event, entry["fn"])`, whose `remove` filters `entry["fn"] != handler` — again nuking both subscriptions of the same function. Its `remove` for plain `unsubscribe` is also remove-all rather than remove-first, though the oracle apparently only punishes the once/persistent interaction.

### Notable passes

- **08 (the discriminator):** The six passers all converged on *entry-identity* removal. fable/opus/sonnet/glm/nemotron added a `remove_entry`/`discard_entry` method that removes by `is` identity, leaving other entries for the same fn intact. kimi took a structurally different route: snapshot the handler list, then `remove_once(event)` strips all once-entries *before* invoking anything — arguably the cleanest for nested-publish safety.
  - Raise-safety split: fable and opus remove the once-entry *before* calling the handler; sonnet, glm, and nemotron use `try/finally`. Both are raise-safe. gemma4's `try/finally` was raise-safe too — its bug was orthogonal.
  - One subtlety I'd flag: fable's and opus's `remove` (plain unsubscribe) removes only the *first* matching entry, while kimi/glm/nemotron/haiku remove *all* matches. The oracle apparently accepts both, so unsubscribe-with-duplicates semantics are under-specified.
- **06:** All passers used lexicographic lock ordering; glm and sonnet wrote the tidiest version (`first, second` swap). fable's branch duplication (full lock/defer in both if-arms) is more verbose but equivalent. sonnet compares account *pointers* (`f == t`) rather than names — marginally more robust if two names ever aliased one account.
- **07:** All eight produced the identical canonical `seen`-set fix. Nobody used `dict.fromkeys` — fine, since the loop preserves order equally well; zero variance here.
- **02:** glm and nemotron used `sync.RWMutex` with read locks on `Balance`/`Total` — slightly more sophisticated; everyone else used a plain `Mutex`. Both correct.
- **05 (refactor):** All behavior-preserving. kimi over-engineered a touch (a `COUPON_RULES` table plus a `_build_result` helper — 7 functions for a 50-line problem); glm has a typo constant `COON_DISCOUNT_AMOUNT`, harmless but sloppy. haiku's and gemma4's decompositions were clean and proportionate. Tier-1 quality differences are cosmetic across the board.
- **03:** Everyone wrote a char-by-char quote state machine. glm alone rewrote the whole function to scan raw text (handling `\r\n` and embedded newlines in quoted fields implicitly dropped by `splitlines`... actually glm's version would *keep* a quoted field's newline as a row break since it processes `\n` unconditionally when `in_quotes` is false — but inside quotes it appends it, so glm actually handles multi-line quoted fields, the only model that does). 6.4k output tokens for it, though — the most expensive tier-1 solution in the run.

## 5. Rankings

- **Best correctness:** six-way tie at 8/8 (kimi, glm, nemotron, sonnet, opus, fable). *Caveat: single sample; the tie means this run can't separate the top six.*
- **Best correctness-per-dollar (Claude only):** sonnet — 8/8 at $1.70, beating opus ($2.06) and fable ($4.63) at equal score. haiku is cheapest ($0.44) but at 6/8 that's false economy. *Caveat: Ollama models are excluded only because they're unmetered, not because they're free to run.*
- **Fastest (wall/task):** gemma4 at 6.7s/task — 3–4× faster than the next group — but it dropped a task. Fastest at 8/8: kimi (23.1s/task). *Caveat: cloud queue variance is huge (nemotron's 08 took 539s); wall time conflates model speed and infrastructure.*
- **Most surgical (edits/task):** gemma4 and haiku (+15.5/-7.8 and +15.5/-8.0 per task), then fable (+16.8/-8.8). *Caveat: on this suite edit counts mostly reflect refactor-style verbosity on 05/08, and the one-shot adapter's whole-file returns are re-diffed against main, so the numbers are more comparable than usual — still a churn signal, not a quality signal.*

## 6. Caveats

- **Single sample at nonzero temperature.** The 8/8 six-way tie and haiku's two misses could shuffle on a re-run; treat rank gaps of one task as noise.
- **The suite has saturated except for two hardened edge cases.** Tasks 01–05 and 07 got 8/8 across the board, with 01 and 07 producing byte-identical diffs from most models — they measure nothing at this model tier. All discrimination came from the 06 self-transfer and 08 same-fn once+persistent oracles added in commit 7988df7. To keep the benchmark useful, tier-2 needs more cases of that kind (e.g. 08 nested publish during publish, unsubscribe-during-publish; 06 transfer racing Deposit).
- **Provider reporting asymmetry** as described: Claude costs are real USD, Ollama is $0 by construction; Claude turn/token counts reflect agentic re-reading; direct comparison of raw totals (e.g. the 5.2M summed input tokens) is meaningless.
- **Oracle limitations I can spot:** 05-god-refactor is graded on behavior, so an empty diff (no refactor at all) would presumably pass — I can't tell from this data whether the oracle checks structure. 08's unsubscribe semantics with duplicate subscriptions (remove-first vs remove-all) are unpinned — both variants passed. 06 doesn't appear to test transfers racing `Deposit` on the same account (Deposit locks only one mutex — that's fine, but nothing verifies conservation under mixed load beyond what the visible tests cover).
- **nemotron's wall time is untrustworthy as a model signal** — 59% of its 918s was one call; without server-side timing you can't separate queueing from inference.
