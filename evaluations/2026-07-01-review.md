# Benchmark Run Review — 2026-07-01

## 1. Executive summary

Five of seven models went 8/8 (kimi-k2.7-code, glm-5.2, sonnet, opus, nemotron-3-ultra); gemma4 scored 7/8 and haiku 6/8. Tier-1 (`01`–`05`) was effectively free — on `01`, `02`, and `07` the diffs are **byte-identical across all seven models** — so the entire discriminating power of this run lives in the two tier-2 concurrency/lifecycle tasks (`06`, `08`). The single most interesting finding: **both failing models tripped on the same subtle bug class in `08`** — cleaning up a once-handler by *function identity* (`entry["fn"] == handler`) instead of *entry identity*, which also deletes a co-registered persistent handler for the same function. Haiku additionally missed the self-transfer reentrant-lock guard in `06`, double-locking a non-reentrant mutex and deadlocking. Net: the hardened oracles (self-transfer, same-fn once+persistent) did exactly their job — they separated the field where the visible happy-path tests could not.

## 3. Solution review

### Failures (root-caused)

- **gemma4 — `08`**: `registry.remove()` filters out **every** entry whose `fn` matches (`entry["fn"] != handler`). When one function is subscribed both `once` and persistently, firing the once-handler calls `unsubscribe(event, fn)`, which removes *both* entries. Third publish sees no persistent handler → `[1, 1]` vs expected `[1, 1, 2]`. Root cause: **remove-by-function-identity instead of remove-by-entry-identity.** (Its `publish` is otherwise fine — `handlers()` returns a `list(...)` copy, so iteration is safe.)
- **haiku — `08`**: Identical root cause — `remove()` filters `entry["fn"] is not handler`, nuking the co-registered persistent handler. Same `[1, 1]` failure.
- **haiku — `06`**: Locks in name order but has **no `from == to` guard**. On a self-transfer `f` and `t` are the same `*account`, so `first == second`, and `first.mu.Lock(); second.mu.Lock()` locks the same non-reentrant `sync.Mutex` twice → deadlock. The watchdog test `TestSelfTransferNoDeadlock` times out at 5s, five times. Every other model special-cased self-transfer (gemma4/glm/kimi/nemotron via `if from == to { return }`; opus/sonnet via single-lock-then-mutate).

### Notable passes on the hard tasks

**`06` (lock ordering).** All passers acquire the two mutexes in a consistent total order by account name — the textbook fix. Two idioms for the self-transfer edge:
- **Early return** (gemma4, glm, kimi, nemotron): `if from == to { return }`. Correct because a self-transfer is net-zero; skipping it is observably equivalent.
- **Single-lock-and-mutate** (opus, sonnet): lock once, still perform `-=/+=` on the same account. Also correct and arguably more faithful (preserves the write path), but more code. Sonnet compares pointers (`f == t`); opus compares names — equivalent given the map.

**`08` (snapshot + raise-safety).** The passers split cleanly by cleanup mechanism — and it's exactly the mechanism the failures got wrong:
- **Remove by entry identity** (opus, sonnet, glm, nemotron): a `remove_entry` that deletes the specific dict via `is`. This is what makes same-fn once+persistent work. Sonnet/nemotron/glm wrap the call in `try/finally` so a raising handler still removes the once-entry (raise-safe); opus removes **before** invoking (also raise-safe, and re-entrancy-safe).
- **Remove-all-once up front** (kimi): snapshots handlers, then `remove_once(event)` strips every once-entry *before* dispatch. Cleanest re-entrancy story (a nested publish inside a handler won't re-fire the once-handlers) and inherently raise-safe. My pick for the most robust `08`.
- Note the discriminator: opus/sonnet's `remove()` also deletes only the **first** matching fn (`return` after one `del`), so even their `unsubscribe` path is safe against the same-fn trap — unlike gemma4/haiku's list-comprehension-filter.

No solution looked over-engineered or risky-but-passing; glm's `02` and nemotron's `05` are the most verbose (RWMutex, per-step helper functions) but idiomatic. nemotron's `04` adds a defensive `start >= total → []` branch that's harmless (slice already handles it).

## 5. Rankings

- **Best correctness:** 5-way tie at 8/8 (kimi, glm, sonnet, opus, nemotron). *Caveat:* single sample at temperature — a rerun could reshuffle the tail.
- **Best correctness-per-dollar (Claude only):** among perfect scorers, **sonnet** ($0.213/task) beats opus ($0.257/task). But **haiku is the cheapest per pass** ($0.074/pass, 13.5 passes/$) despite going 6/8 — cheap enough that a retry-on-fail wrapper could be competitive. *Caveat:* Ollama models are unmetered, so this ranking is Claude-internal only.
- **Fastest (wall/task):** gemma4 (6.6s) — but it's 7/8. Among 8/8 models, **kimi** (23.1s/task) is fastest; nemotron (114.8s) is an order slower purely from cold-start. *Caveat:* cloud queue variance dominates, not model speed.
- **Most surgical (edits/task):** among 8/8, **opus** (+143/−67 ≈ 26/task) edges sonnet (+145/−68); glm is the churniest (+197/−74 ≈ 34/task). *Caveat:* edit count is a churn signal, not correctness, and every model touched the same 9 files, so the spread is small and largely stylistic here.

## 6. Caveats

- **Single sample + temperature** ⇒ run-to-run variance is real; the 8/8 cluster and especially the haiku/gemma4 failures could flip on a rerun. Don't over-read one run.
- **Provider reporting asymmetry:** Claude reports real `cost_usd` and inflates input via cached re-reads across many turns; Ollama reports `cost_usd == 0` (unmetered), one call/task, and whole-file outputs. Never compare raw input/cost/edit numbers across the two.
- **Whole-file vs surgical:** Ollama emits entire files, structurally favoring Claude on edit-count minimalism — treat edits as churn, not skill.
- **Oracle limitations I noticed:**
  - `07` (perf-dedup): the *original* code is already functionally correct — just O(n²). A model that changed nothing would pass any purely functional test. Everyone applied the `seen = set()` fix, so I can't tell from this data whether the oracle actually times large input or just got lucky that all models fixed it; correctness rests on the oracle being a real perf gate.
  - `06` and `08` prove the discriminating tests are the *hardened* ones (self-transfer, same-fn once+persistent) — the visible happy-path behavior passes for gemma4 and haiku too. The benchmark's signal is entirely in those two hidden edge cases; without them this run would report a meaningless 7-way 8/8 tie.
