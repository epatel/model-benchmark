# Benchmark run review — 2026-07-01

## 1. Executive summary

Five of eight models went 9/9: sonnet, opus, fable, glm-5.2_cloud, and nemotron-3-ultra_cloud. gemma4_cloud and kimi-k2.7-code_cloud each dropped one task; haiku was the outlier at 6/9, failing the two classic edge-case traps (CSV trailing comma, self-transfer deadlock) plus the event-bus once/persistent test. Also note the prompt says 8 tasks but this run has 9 (`09-glob-matcher`, tier-3), which everyone passed — it discriminated on approach quality, not correctness.

## 3. Solution review

### Failures

- **haiku / kimi — 03-csv-parser** (identical root cause): both wrote a field-at-a-time loop that *consumes* a trailing comma after the last field but never emits the empty field that follows it, so `"a,b,"` → `[a, b]`. The models that passed either used a char-level state machine that unconditionally appends the in-progress field after the loop (fable, opus, sonnet — trailing empty falls out naturally) or bolted on an explicit patch (nemotron's `if line.endswith(',')`, glm's inline `if i == n: fields.append('')`). Classic "correct on visible tests, wrong on the fencepost" failure.
- **haiku — 06-deadlock**: fixed the opposing-transfer deadlock with name-ordered locking but ignored `from == to`. With `from <= to`, `first` and `second` are the *same* `*account`, so it locks the same mutex twice → guaranteed self-deadlock, exactly what `TestSelfTransferNoDeadlock` watchdogs (5s × 5 repeats). Incomplete fix, not a wrong idea.
- **haiku / gemma4 — 08-event-bus** (identical root cause): after a once-handler fires, both call `remove(event, fn)` where `remove` strips **all** entries whose `fn` matches. When the same function is subscribed both once and persistently, the persistent subscription is destroyed too → second publish fires nothing, `[1,1]` instead of `[1,1,2]`. The passing models removed by *entry identity* (fable, opus, glm, nemotron) or stripped once-entries from the registry before invoking the snapshot (kimi).

### Notable passes

- **08 (the real differentiator):** kimi's is arguably the best design — snapshot handlers, `remove_once` *before* invocation, so once-semantics survive raising handlers *and* mid-publish subscriptions. fable removes the once-entry before calling it (raise-safe) but without `try/finally`; opus/glm/nemotron use `try/finally` + entry-identity removal — all sound. **sonnet's** `remove(event, entry["fn"])` removes only the *first* fn-match; it produces `[1,1,2]` but if the persistent subscription was registered first, the persistent entry is what got deleted, and the surviving once-entry impersonates it for exactly one more publish. I can't see the hidden test's subscription order, but this is a pass the oracle can't distinguish from a bug — see caveats. nemotron reaches into `registry._remove_entry` (private-by-convention it just invented — cosmetic).
- **06 lock ordering:** all passers used consistent ordering; they differ on style. gemma4 early-returns on `from == to` without locking (fine — a no-op transfer); fable/sonnet/opus/glm lock once and proceed; **kimi over-engineered** it, threading a new `id` field through `account` and `New()` when comparing the name strings (which everyone else did) suffices. glm's self-transfer branch does `balance -= amount; balance += amount` — a deliberate no-op that reads oddly but is harmless.
- **07:** all eight produced the same 4-line `seen`-set fix. Zero signal.
- **09-glob-matcher:** the widest quality spread of the run. fable, opus, and sonnet wrote the optimal iterative single-star-backtracking matcher (O(n·m), no allocation per state); sonnet added a clean tokenizer on top. haiku took the low-effort path — bolt a memo dict onto the existing recursion (correct, O(n·m), but keeps Python recursion depth as a liability). gemma4, glm, nemotron, and kimi all wrote full DP rewrites; kimi's is a 99-line token-table + bytearray-rows affair — correct but the most machinery for the same result.
- **gemma4 — 03-csv-parser is a red flag despite passing**: the diff leaves **two definitions of `parse_csv`** in the file, the first a ~40-line abandoned draft (with its own dead re-evaluation commentary) that the second definition silently shadows. It passes only because Python takes the last `def`. That's not a solution, it's an unedited transcript of thinking.
- **02:** everyone added a coarse mutex; gemma4/haiku/nemotron used `RWMutex` with `RLock` on reads — marginally nicer, equally correct at this scale.
- **05:** all structurally similar helper decompositions. fable/opus/sonnet also lifted magic numbers into named constants; haiku's `_build_result` helper is decomposition past the point of usefulness.

## 5. Rankings

- **Best correctness:** 5-way tie at 9/9. On solution *quality* within the tie: opus and fable (surgical, idiomatic, raise-safe 08, optimal 09) edge out sonnet (fragile 08 removal), glm (solid throughout), and nemotron (correct but reaches into a private helper in 08). *Caveat: one sample; the tie could break either way on a rerun.*
- **Best correctness-per-dollar (Claude only):** sonnet — 9/9 at $1.87 ($0.208/pass), vs opus $0.276 and fable $0.617 for the same score. Haiku is cheapest per pass ($0.088) but 3 failures make that a false economy. *Caveat: costs include cache pricing dynamics that vary with runner config.*
- **Fastest (wall/task):** gemma4_cloud at 8s/task — but it's the smallest-output model and dropped a task; among 9/9 models, sonnet (20.9s/task). *Caveat: cloud queue/cold-start noise is visible in this very data (gemma's 34s task 07).*
- **Most surgical (edits/task):** fable and opus (~+18 added lines/task, targeted diffs). *Caveat: metric structurally favors agentic editors over one-shot whole-file emitters.*

## 6. Caveats

- **Single sample at nonzero temperature.** haiku's 3 failures are all "missed edge case," one hidden test each — plausibly 8/9 or 5/9 on a rerun. Same for gemma4 and kimi's single misses.
- **Prompt/data mismatch:** the review brief describes 8 tasks; the run has 9. All numbers here use n_tasks = 9.
- **Oracle blind spot in 08:** the hidden test apparently stops after two publishes. Sonnet's remove-first-match-by-fn fix passes, but depending on subscription order it deletes the *persistent* entry and lets the once-entry stand in for it — a third publish (persistent should still fire) would likely distinguish correct fixes (fable/opus/glm/nemotron/kimi) from sonnet's. Worth adding to the oracle.
- **Oracle can't see code quality:** gemma4's 03 solution ships a dead duplicate `parse_csv` definition and still passes; nemotron's 03 passes via an `endswith(',')` bolt-on. A lint/cleanliness gate or human spot-check would catch these.
- **Tasks 01 and 07 are saturated** (8 near-identical diffs) — they add no signal and mostly pad pass rates.
- **Provider asymmetry** as normalized above: Ollama cost is unmetered (no USD ranking possible), output tokens are inflated by whole-file emission, and Claude input is dominated by cheap cache reads — headline token totals are meaningless without the split.
