# Benchmark run review — 2026-07-10 (n_tasks = 10)

## 1. Executive summary

Four models went 10/10: **gpt-5.4-cc, gpt-5.5-cc, and kimi-k2.7-code_cloud in both modes** — kimi is the only model that was perfect both agentically and one-shot. The most interesting finding is a clean provider-family blind spot: **every Anthropic model (fable, opus, sonnet, haiku) failed `06-deadlock` the same way** — all applied the textbook lock-ordering fix but none guarded the self-transfer case, so `Transfer(a, a)` locks the same non-reentrant mutex twice; every GPT and kimi variant added the `from == to` guard. Meanwhile fable, the most expensive model in the run ($6.09), tied sonnet at 8/10 — cost and correctness were essentially uncorrelated. A secondary flag: the three one-shot failures on `10-txn-store` (gpt-5.4, gpt-5.5, glm-5.2_cloud) reproduce the *unmodified baseline's* failures and have no diff in the evaluation, which smells like a patch-apply/harness failure rather than a reasoning failure.

## 3. Solution review

**Failures, root-caused:**

- **06-deadlock — fable, opus, sonnet, haiku, glm-5.2_cloud-cc (all identical failure mode).** Each replaced the naive lock order with sort-by-name ordering, which correctly fixes the opposing-transfer deadlock the visible code hints at. None handled `from == to`: when the names are equal, `first` and `second` alias the same account and the code locks one `sync.Mutex` twice → guaranteed deadlock, caught by the hidden `TestSelfTransferNoDeadlock`. Haiku's `if from < to { … } else { … }` has the same hole (equal names fall into the else branch and still double-lock). This is "fixed the bug they were shown, not the invariant" — and it split exactly on model family.
- **08-event-bus — fable, sonnet, haiku (first-match removal); gemma4_cloud (remove-all).** The hidden test subscribes the *same function* both persistently and once. The failing models remove the fired once-entry via `remove(event, entry["fn"])`, which matches by function and deletes the *first* entry — the persistent one. Result `[1, 1, 2]` instead of `[1, 1, 2, 3]`. gemma4_cloud's one-shot `remove` filters out *all* entries for the fn, killing both subscriptions at once (`[1, 1]`, two tests fail). The passers all realized identity matters: they added `remove_entry` (removal by entry object identity — glm, gpt-5.4/5.5-cc, gpt-5.6-luna, opus) or filtered by the `once` flag (kimi-cc) or bulk-removed once-entries before dispatch (kimi one-shot). This task was the sharpest discriminator of actual design care in the run.
- **10-txn-store — gpt-5.4, gpt-5.5, glm-5.2_cloud: probable harness artifact.** All three fail with the baseline's exact signature (`begin() copies the store` + ABA conflict not raised), the combined evaluation contains **no diff for any of them on this task**, and glm's usage JSON shows 0 input/0 output tokens for the project. gpt-5.4/5.5 did emit tokens (713 and 2,542), so their patches likely failed to apply. I can't distinguish "model produced an unappliable patch" from "runner bug" from this data — worth re-running before counting these as model failures.
- **10-txn-store — gpt-5.6-luna (real failure).** Its `commit()` unconditionally calls `_commit`, which validates the read-set *before* the early-return for empty write-sets — so a read-only transaction whose read key was concurrently modified raises Conflict, violating the "read-only never conflicts" spec. One line of ordering away from correct.
- **10-txn-store — haiku (real failure).** Its "snapshot" is lazy: `get()` caches `store._data.get(key)` at *first read time*, not begin time. Reads that happen after a concurrent commit see post-commit state — not a snapshot at all. Both hidden snapshot tests fail.

**Notable passes:**

- **06 lock ordering:** gpt-5.5 was the most thorough — sorted-name ordering, self-transfer guard, *and* a rewritten `Total()` that locks all accounts in order (the only model to notice `Total()` can observe a half-completed transfer). kimi one-shot took a distinctive route: assigned each account an integer `id` and ordered by that. gpt-5.4's self-transfer branch (`f.balance -= amount; f.balance += amount`) is a no-op written out literally — odd but harmless.
- **07:** all 15 models used the `seen`-set idiom; nobody used `dict.fromkeys`. Zero variance.
- **08 raise-safety:** most passers wrapped the handler call in `try/finally` to remove the once-entry even on exception; opus and gpt-5.6-luna removed *before* firing, which is the cleaner invariant ("registered at most once per publish, no matter what the handler does"). kimi one-shot's snapshot-then-`remove_once` bulk approach is the most elegant.
- **09 glob:** three distinct correct strategies — memoization (haiku, gpt-5.4/5.5-cc, glm-cc: minimal, surgical), bottom-up DP tables (sonnet, kimi, gpt-5.5, glm one-shot), and the greedy two-pointer star-backtracking algorithm (fable, opus: the tightest asymptotics and the smallest diff of the non-memo group). gemma4_cloud's one-shot DP is correct but carries dead code and an awkward trailing-comma special case in its 03 parser too — passing but sloppy.
- **10 MVCC:** the strong passers (fable, sonnet, glm-cc, gpt-5.4/5.5-cc, kimi both) converged on the same design: global commit clock, per-key version history, read-set tracking, version-based (not value-based) conflict detection — which is exactly what the ABA test demands. **Opus's pass is the risky outlier:** instead of versioned history it keeps a registry of *active transactions* and has committers push begin-time values into other open transactions' `_saved` maps before overwriting. It passes, but it's O(active-txns) per commit, holds references to every open transaction (leak if a txn is dropped without commit/abort), and is far harder to reason about than the history approach every other passer used. Risky-but-passing.

## 5. Rankings

- **Best correctness:** gpt-5.4-cc, gpt-5.5-cc, kimi-k2.7-code_cloud (×2), all 10/10. Kimi gets the edge for doing it in both modes — one-shot 10/10 with no test feedback loop is the strongest single result here. *Caveat:* single sample; the gap to the 9/10 cluster is one task.
- **Best correctness-per-dollar (Claude only):** haiku at $0.090/pass — but it's also the least correct (7/10). Among models you'd actually trust with tier-2+, **opus** ($0.330/pass, 9/10) is the pick; fable is strictly dominated (2× opus's cost, one fewer pass). *Caveat:* Ollama models are excluded as unmetered, and any output-token price estimate would flip this ranking.
- **Fastest:** gpt-5.4 one-shot, 3.6s/task at 9/10. *Caveat:* one-shot has no tool round-trips, and cloud queue variance is invisible in a single run (compare glm one-shot's 361s on a task where it emitted zero tokens).
- **Most surgical:** gemma4_cloud-cc (+16.3/-9.4 lines per task) and fable (+16.8/-9.7). *Caveat:* churn ≠ quality — gemma4-cc failed 08, and gpt-5.5's larger 06 diff is larger precisely because it also fixed `Total()`.

## 6. Caveats

- **Single sample at nonzero temperature.** Every 9-vs-10 distinction here is one task; opus vs sonnet vs fable orderings could easily flip on a re-run. The family-level 06 pattern (4/4 Anthropic failures with identical root cause) is the only result robust enough to act on.
- **Probable harness artifact on task 10.** Three one-shot models (gpt-5.4, gpt-5.5, glm-5.2_cloud) show baseline-identical failures with no applied diff; glm additionally reports 0 tokens for the project. Their "9/10" may understate them — re-run before drawing conclusions, and consider making the runner fail loudly when a patch doesn't apply.
- **Provider reporting asymmetry.** Ollama cost is unmetered, the `-cc` adapters report no cache split (their multi-million input totals aren't comparable to Claude's cached reads), and Ollama output likely bundles reasoning tokens (kimi's 64.9k outlier).
- **Saturated tasks.** 01, 02, 04, 05, 07 were solved by all 15 models, several with byte-identical diffs — half the suite now measures nothing. The signal lives entirely in 06/08/10.
- **Oracle blind spots.** 05 accepts any behavior-preserving shuffle, so it can't distinguish a good decomposition from a cosmetic one. On 06, the hidden suite apparently doesn't fail gpt-5.5's peers for the racy `Total()` it alone fixed — meaning a torn-read bug passes the oracle. And opus's task-10 design passes despite leaking active-transaction references; the oracle checks semantics, not resource hygiene.
