# Benchmark run review — 2026-07-10 (n_tasks = 9)

## 1. Executive summary

Eight of sixteen entries went 9/9; the run's discriminators were the two tier-2 concurrency-ish tasks, **06-deadlock** (5 failures) and **08-event-bus** (5 failures). The most interesting finding: **every Anthropic model in the run (opus, sonnet, haiku, fable) failed the hidden self-transfer test on 06** — all four applied textbook lock-ordering by account name but none guarded `from == to`, so a self-transfer double-locks the same non-reentrant mutex — while most OpenAI/Ollama entries added the guard. The `gpt-5.6-luna-cc` 2/9 result is a **harness failure, not a model failure**: its usage JSON shows ~1.4k output tokens total, one task with literally zero tokens, and a total diff of `+1/-1 (1f)` — it made exactly one edit all run, and the same model one-shot scored 9/9. That artifact also exposes an oracle hole: it "passed" 05-god-refactor **without changing a line**, because a behavior-preserving refactor task cannot be graded by behavior alone.

## 3. Solution review

### Failures, root-caused

**06-deadlock — fable, opus, sonnet, haiku, glm-5.2_cloud-cc (identical bug).** All five wrote the same by-name lock-ordering fix (`first, second := f, t; if from > to { swap }`) with no same-account check. For `from == to`, `first` and `second` alias the same account, and the second `Lock()` on Go's non-reentrant `sync.Mutex` self-deadlocks. The fix passes the *visible* opposing-transfers test; only the hidden self-transfer test catches it. This is "correct pattern, missed edge case" — the fix is one `if from == to` guard away, which the passers (gpt-5.4, gpt-5.5, both gemmas, both kimis, glm one-shot, gpt-5.5-cc/5.4-cc, luna one-shot) all added. Notably glm one-shot added the guard while glm-cc didn't — same weights, different run, single-sample noise.

**08-event-bus — fable, sonnet, haiku (identical bug).** All remove a fired once-handler by *function equality* (`remove(event, entry["fn"])`), which deletes the **first** registry entry matching that fn. When the same function is subscribed both once and persistently, the persistent entry (subscribed first) gets removed instead of the once entry — calls go `[1, 1, 2]` instead of `[1, 1, 2, 3]`. The passers removed by **entry identity** (`remove_entry`, opus/gpt-5.4/5.5/glm) or by a once-flag filter (kimi variants). Fable and sonnet had the concept (snapshot + removal) but the wrong removal key. Also note fable removes *before* firing (raise-safe) while sonnet uses `try/finally` (also raise-safe); both still fail on the identity issue.

**08 — gemma4_cloud (worse variant).** Its `remove` deletes **all** entries with a matching fn, so both the once and the persistent subscription die on first publish (`[1, 1]`, two hidden tests fail). gemma4_cloud-cc used first-match removal and fails only the ordering case. Same family, same conceptual blind spot at two severities.

**gpt-5.6-luna-cc — everything.** The reproduced failures are the *unmodified* baseline failing: 03 shows naive `split(",")` output, 04 returns all 25 users, 08 raises the stub's `NotImplementedError`, 09 is still exponential, 02 races on the bare map. Total edits: `+1/-1` in one file (the 01 LRU fix). Usage confirms it: 69–360 output tokens on most tasks, and **0 input/0 output tokens on 07** — the bridge dropped or truncated responses. Given luna one-shot went 9/9 and the repo history includes a FILE-marker-parsing bug that previously zeroed out gpt-5.4/5.5 runs, this is almost certainly the same adapter class of failure. This row should be excluded from model comparisons.

### Notable passes

- **01, 07** are effectively non-discriminating: all 16 entries produced near-byte-identical fixes (drop the `+ 1`; add a `seen` set). Nobody used `dict.fromkeys` on 07; the seen-set version is arguably better anyway since it preserves the explicit loop. The uniformity on 01 across providers and temperatures is striking — the task admits exactly one obvious one-token fix.
- **06 quality among passers**: gpt-5.4's self-transfer branch does `f.balance -= amount; f.balance += amount` under one lock — semantically faithful to "a transfer happened" and harmless. gpt-5.5 went furthest: sorted-name slice on the Bank plus lock-*all-accounts* `Total()` — that makes `Total()` a linearizable snapshot rather than the original's eventually-consistent sum. Robust, but it changes concurrency behavior beyond the bug and serializes all transfers against every Total call; I'd call it over-engineered-but-defensible. kimi one-shot invented integer account ids for ordering — more mechanism than needed when names already order.
- **08 quality among passers**: kimi one-shot's design is the cleanest conceptually — snapshot handlers, strip all once-entries *before* invoking, then fire the snapshot; raise-safety falls out for free. opus removes each once-entry by identity before firing with a comment explaining why. glm/gpt-5.4/5.5 use `try/finally` + identity removal — equivalent. kimi-cc's `remove(event, handler, once=True)` overloads the public-ish remove signature; works but is the least tidy.
- **09 approaches** split three ways, all passing: (a) *memoize the existing recursion* (haiku, gpt-5.4, gpt-5.4-cc, gpt-5.5-cc, glm-cc, gemma4-cc) — lowest-risk, exact semantics trivially preserved; (b) *iterative greedy star-backtracking* (fable, opus) — classic O(n·m), most surgical of the real rewrites; (c) *full DP rewrite with pattern tokenization* (sonnet, gpt-5.5, gpt-5.6-luna, glm, both kimis, gemma4) — correct here, but the riskiest class since the unterminated-`[` edge case had to be re-derived; gemma4_cloud's version is sprawling (reimplements class parsing inline, keeps the old `_match_class` "for compatibility"). For a task graded on *exact* semantics preservation, memoization was the smart play.
- **02**: gpt-5.5 added `if l.balances == nil` guards in Deposit/Transfer — dead code given `New()` always initializes the map; harmless over-defensiveness.
- **05**: everyone converged on the same subtotal → discount → coupon → shipping → tax decomposition; kimi-cc's `_round_money` wrapper and gpt-5.5's coupon-name constants are mild over-abstraction. haiku's `coupon_discounts = {"SAVE5": -5}` dict-of-negative-numbers is the one slightly odd construction, though it works.

## 5. Rankings

- **Best correctness:** 8-way tie at 9/9; among them **gpt-5.4** is the standout (perfect in 29s wall-clock with the smallest output volume). *Caveat: single sample; five of the ties are separated only by tasks 06/08, which coin-flip on one edge case.*
- **Best correctness-per-dollar (Claude only):** **haiku** — 7 passes / $0.52 ≈ 13.5 passes/$ vs sonnet 3.7, opus 3.4, fable 1.3. *Caveat: haiku, sonnet, and fable failed the identical two hidden tests, so within-Claude "correctness" differences here are one task (opus's 08 pass) at 4–10× the price.*
- **Fastest (wall/task):** **gpt-5.4** at ~3.2 s/task (gemma4_cloud ~3.4). *Caveat: one-shot cloud calls vs agentic tool round-trips aren't the same execution model; cloud queue variance unknown.*
- **Most surgical (edits/task):** excluding the broken luna-cc run, **gemma4_cloud-cc** (+131/−82 over 10 files), then fable (+145/−86). *Caveat: churn ≠ quality — gemma4-cc's low churn includes the buggy 08 fix, and one-shot whole-file rewrites structurally inflate the other group's counts.*

## 6. Caveats

- **Single sample at nonzero temperature.** glm passed 06 one-shot and failed it via the cc bridge with an otherwise near-identical diff; that alone shows one run cannot separate 8/9 from 9/9 models. The 06/08 failure clusters are more trustworthy than any individual rank.
- **The gpt-5.6-luna-cc row is a harness artifact** (one edit all run, one task with zero tokens) and should be excluded, not reported as a model score. Given the repo's prior FILE-marker parsing bug zeroed other runs, the openai-cc adapter deserves a health check (e.g., fail the run if a task ends with an empty diff *and* the stub tests still raise NotImplementedError).
- **Oracle limitation, demonstrated in-run:** 05-god-refactor was "passed" with zero changes by the broken run. A behavior-preserving refactor task graded by behavioral tests cannot fail a no-op; it needs a structural check (function count, cyclomatic complexity, or a diff-nonempty gate).
- **Low discriminative power in tier-1:** 01 and 07 produced essentially identical diffs from all 16 entries; they measure "can the model make the canonical one-line fix," which everything above the broken run can.
- **Provider reporting asymmetry:** Ollama rows carry no cost and the cc-bridge rows report no cache split, so their 1.4–2.5M "input tokens" are raw resend volume, not comparable to Claude's fresh-token figures. All cross-group efficiency comparisons above are indicative only.
- **06's hidden oracle is good** — it caught a real, subtle bug in five submissions — but note all five failures are the *same* bug; the task effectively measures one binary edge-case check, so its 5-failure count overstates how much signal it adds beyond "did you think about self-transfer."
