# Run review — 2026-07-10 (n_tasks = 9)

## 1. Executive summary

Seven of fourteen models went 9/9: gpt-5.4, gpt-5.5, and kimi-k2.7-code in **both** one-shot and agentic (`-cc`) modes, plus glm-5.2 one-shot. All four Claude models failed at least one tier-2 task, and the Anthropic-billed group finished 8/9 (opus) and 7/9 (sonnet, haiku, fable) — with fable the most expensive model in the run ($5.26) tied for last place. The single most interesting finding: **every 06-deadlock failure is the identical mistake** — models correctly pattern-matched "lock ordering by name" for the visible opposing-transfer deadlock but missed that a self-transfer (`from == to`) makes `first` and `second` the *same* mutex, which is then locked twice. Similarly, all five 08-event-bus failures share one root cause: removing a fired once-handler *by function* instead of *by entry identity*, which deletes the wrong subscription when the same function is registered both once and persistently. Tier-1 and 07/09 discriminated nothing — 14/14 passed all of them, with several byte-identical diffs.

## 3. Solution review

### Failures

**06-deadlock — fable, opus, sonnet, haiku, glm-5.2_cloud-cc — all the same bug.** Each replaced the naive `f.lock; t.lock` with lock ordering by account name, which fixes the visible opposing-transfer deadlock. None guarded `from == to`, so on a self-transfer `first` and `second` alias the same account and the code locks one `sync.Mutex` twice → `TestSelfTransferNoDeadlock` times out at 5s. Haiku's `if from < to … else …` variant has the identical hole (equality falls into the else branch and double-locks). The hidden test repro confirms: deadlock on self-transfer, five retries each. Notably all four Claude models in the run share this blind spot; every passer added an explicit `from == to` (or `f == t`) early-return or single-lock branch.

**08-event-bus — fable, sonnet, haiku, gemma4_cloud-cc — remove-by-function instead of remove-by-entry.** The hidden test subscribes the *same function* both `subscribe_once` and `subscribe` (persistent), expecting `[1, 1, 2, 3]`. These models remove the fired once-entry via `registry.remove(event, entry["fn"])`, which deletes the **first** entry matching that function — the persistent one. The once-entry survives (still flagged `once`), fires on publish 2, then removes itself → `[1, 1, 2]`. **gemma4_cloud (one-shot)** is worse: its `remove` filters out *all* entries for the function, so both subscriptions die after publish 1 → `[1, 1]` and two test failures. Passers either added an identity-based `remove_entry` (opus, glm-5.2 both modes, all three gpt runs), a `once=`-filtered remove (kimi-cc), or stripped once-entries from the registry before invoking the snapshot (kimi one-shot).

Note fable's 08 diff has a second latent issue the oracle didn't reach: it removes the once-entry *before* calling but its `remove` is by-fn, and it also skips the try/finally raise-safety most passers added — the by-fn removal alone was fatal here.

### Notable passes

- **06 lock ordering:** gpt-5.5 was the most thorough — self-transfer branch, ordered locking, *and* a rewrite of `Total()` to lock all accounts in sorted order (arguably over-engineered; the original per-account lock-read-unlock in `Total` wasn't under test, and this change trades away concurrency). kimi one-shot took a distinctive route: stable integer `id` per account and ordering by id — robust and avoids string comparison, at the cost of a struct change. gpt-5.4's self-transfer branch does `balance -= amount; balance += amount` — a deliberate no-op that's faithful to "what a transfer does" but reads oddly.
- **07 perf-dedup:** literally all 14 models produced the same seen-set fix (several byte-identical). Nobody used `dict.fromkeys`; no discrimination value.
- **08 snapshot + raise-safety:** the cleanest solutions (gpt-5.4/5.5 both modes, glm-5.2 both modes) combine `try/finally` (once fires exactly once even if it raises) with identity-based removal. kimi one-shot's remove-before-invoke achieves the same exactly-once guarantee without try/finally and is arguably the most elegant. opus removes-before-firing with identity removal — also correct.
- **09 glob-matcher:** the widest spread of approaches, all passing. Memoized recursion (gpt-5.4, gpt-5.5-cc, haiku, glm-5.2-cc, gemma4-cc) is the minimal-diff route. Full bottom-up DP (sonnet, kimi both modes, glm-5.2, gpt-5.5 with a token pre-parse) is heavier but O(n·m) worst case with no recursion-depth risk. fable and opus wrote the iterative greedy two-pointer with single-star backtracking — the most elegant and lowest-memory, and correct here because every non-`*` token consumes exactly one character. Two smells: gpt-5.4-cc's module-level `lru_cache` on `_parse_class(p, pi)` grows unboundedly across calls (fine for tests, a leak in a library), and gemma4_cloud one-shot rewrote `match` as a sprawling forward DP that *duplicates* the class-parsing logic inline while keeping `_match_class` "for compatibility" — passing but the least maintainable diff in the task.
- **02/04/05 (tier-1):** near-uniform. gpt-5.5's nil-map guards in 02 are defensive noise (the constructor always initializes the map). haiku's 04 doesn't clamp `page/limit` to ≥1 (others use `Math.max(1, …)`) — passes, but `limit=0` would divide-by-zero `totalPages` to `Infinity`; the oracle apparently doesn't probe that.

## 5. Rankings

- **Best correctness:** 7-way tie at 9/9. Among them **gpt-5.4** dominates the tie-break — fastest (29s total) and lowest output per task in both its runs. *Caveat: single sample; the tie-breakers measure style, not correctness.*
- **Best correctness-per-dollar (Claude only):** **haiku** — 7 passes for $0.52 (~13.5 passes/$), vs sonnet 3.7, opus 3.4, fable 1.3. *Caveat: only four metered models; haiku's cheapness comes with the same two tier-2 failures as fable at 10× the price.*
- **Fastest (wall/task):** **gpt-5.4 one-shot** (3.2 s/task), gemma4 one-shot right behind (3.4). *Caveat: one-shot cloud calls carry no tool round-trips; queue/cold-start variance makes cross-run wall-clock noisy (gemma4-cc took 86.5 s/task on the same weights).*
- **Most surgical (edits/task):** **gemma4-cc** (+14.6/task) then fable (+16.1). *Caveat: pure churn signal — gemma4-cc's small diffs include a failing 08, and one-shot whole-file returns don't inflate this particular metric since edits are measured as diff-vs-main.*

## 6. Caveats

- **Single sample at nonzero temperature.** The 9/9-vs-7/9 gaps hinge on exactly two hidden edge cases; one different sample on 06 or 08 reshuffles the board. The many byte-identical tier-1 diffs also suggest those seven tasks have near-zero discrimination power — the benchmark is effectively a 2-task test (06, 08) for frontier models.
- **Three reporting regimes, not two.** The `-cc` bridge runs are agentic but unmetered *and uncached* — their 1.5–2.5M fresh input tokens make any "cost" comparison against Claude's cache-dominated traffic meaningless without an assumed price, which I have not applied.
- **Oracle limitations I can spot:** 05-god-refactor grades only behavior preservation — a zero-line "refactor" would pass, so it measures nothing about refactor quality. 06 could be passed trivially by one global bank mutex (serializing all transfers); the oracle shown can't detect lost parallelism. 02 likewise accepts a single coarse lock (which is what everyone wrote). 04's oracle apparently doesn't probe `limit=0`/negative values (haiku's unclamped version passed). 09's semantics tests are strong, but I can't tell from this data whether the polynomial-time requirement is enforced by a timing test or just implied.
- **Self-transfer wasn't hinted.** All five 06 failures fixed exactly the bug that's visible in the code and comment ("opposing transfers"). Whether the benchmark intends "did you think of `from == to`?" as skill or as trivia is worth deciding before weighting tier-2 heavily — though the correlation (all Claude models missed it, all OpenAI/kimi/gemma runs caught it) is a real signal in this run.
