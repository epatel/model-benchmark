# Run review — 2026-07-10, n_tasks = 9

## 1. Executive summary

Eight models went 9/9 — all three GPT-5.x one-shots, gpt-5.6-luna, kimi-k2.7, glm-5.2 (one-shot), and the gpt-5.4/5.5/kimi agentic (`-cc`) variants — while **all four Anthropic models failed 06-deadlock, every one for the identical reason: correct lock-ordering fix with no self-transfer guard**, so `Transfer(a, a, n)` locks the same mutex twice. That's the single most interesting finding: it isn't a capability gradient (fable, opus, sonnet, haiku all wrote essentially the same diff) but a shared blind spot, and it means the priciest model in the run (fable, $5.26) tied for last at 7/9. The second cluster is 08-event-bus, where five models removed once-handlers **by function equality instead of entry identity**, deleting the persistent subscription when the same fn is registered both ways. Task 01 was a one-token fix that all 15 models solved byte-identically — it discriminates nothing. Single run, so treat rank gaps of one task as noise.

## 3. Solution review

### Failures — root causes

**06-deadlock (all five failures are the same bug).** Every failing diff sorts the two accounts by name and locks in order — which fixes the opposing-transfer deadlock the visible symptom suggests — but never handles `from == to`. Haiku's version is the clearest: `if from < to {...} else {...}` — a self-transfer falls into the else branch and locks the *same* mutex twice → guaranteed deadlock, caught by `TestSelfTransferNoDeadlock` (5s watchdog, ×5). Fable, opus, sonnet, and glm-5.2_cloud-cc have the identical hole via `first, second := f, t; if from > to {swap}` — when `from == to`, `first` and `second` alias one mutex. Classified: correct fix for the wrong (incomplete) failure model — they fixed the deadlock named in the task title and stopped. Interesting cross-runner inconsistency: glm-5.2 *one-shot* handled self-transfer explicitly and passed; its agentic `-cc` run missed it.

**08-event-bus — remove-by-fn instead of remove-by-entry.** The hidden test subscribes the *same function* both `subscribe_once` and `subscribe`, expecting `[1, 1, 2, 3]`. Fable, sonnet, haiku, and gemma4_cloud-cc implement once-cleanup as `registry.remove(event, entry["fn"])`, which deletes the **first** entry matching the fn — the persistent one, subscribed earlier. After publish 1, what survives is the *once* entry; publish 2 fires it and removes it; publish 3 fires nothing → `[1, 1, 2]`. gemma4_cloud is worse: its `remove` filters out **all** entries for the fn, so both die on publish 1 → `[1, 1]` and it also fails the reversed-order variant (2 test failures). Note fable removed the once-entry *before* invoking (good raise-safety instinct) but still keyed removal by fn — right ordering, wrong identity.

### Notable passes

- **06 lock ordering**: gpt-5.4 and gpt-5.6-luna special-case `f == t` (pointer compare — also robust to aliasing) with a single lock; gemma4 variants early-return on `from == to` (a no-op self-transfer — observationally equivalent, arguably skips the lock the doc implies). kimi one-shot added an `id int` field to accounts and orders by id — more invasive than needed when names order fine. **gpt-5.5 over-engineered**: it also rewrote `Total()` to lock *all* accounts in sorted order (a consistent snapshot the task didn't ask for) and added nil-map lazy init to the *other* ledger — correct but sprawling, ~3× the diff of peers.
- **07 perf-dedup**: literally all 15 models wrote the same `seen = set()` guard. Zero signal. Nobody used `dict.fromkeys` — arguably the more idiomatic one-liner, but the surgical loop edit is defensible.
- **08 (passers)**: the clean pattern — used by opus, gpt-5.4/5.5, glm-5.2, gpt-5.6-luna — is a dedicated `remove_entry` that removes by **object identity** (`is`), leaving `remove(fn)` for public unsubscribe. kimi one-shot took a different but valid route: snapshot handlers, then `remove_once(event)` *before* invoking anything — once-handlers fire exactly once even if one raises. kimi-cc's `remove(event, handler, once=True)` flag is a slightly leaky API but correct. Most passers wrapped invocation in `try/finally`; opus and gpt-5.6-luna instead remove-before-fire — both satisfy the oracle.
- **09 glob-matcher**: the widest approach spread in the run. Fable and opus wrote the iterative greedy two-pointer with single-star backtracking (the tightest solution, O(n·m), ~30 lines) — high-quality, and identical in strategy, which again suggests a family prior. Most others memoized the existing recursion (haiku, gpt-5.4, gpt-5.5-cc, glm-cc — minimal, safe, preserves semantics by construction) or wrote bottom-up DP (sonnet, kimi variants, glm one-shot). gemma4_cloud and gpt-5.6-luna did full forward-DP rewrites with hand-rolled class re-parsing — the riskiest approach for the "exact semantics preserved" requirement (unterminated-`[` edge case duplicated in new code), but both got it right. gemma4_cloud-cc's `lru_cache(None)` closure is elegant but caches per-call with no leak — fine.
- Tier-1 quality: 01 diffs are 15× byte-identical (one-token fix). 02 splits into `sync.Mutex` (fable, opus, sonnet, kimi one-shot) vs `RWMutex` (everyone else) — RWMutex is premature here but harmless; gpt-5.5 again gold-plated with nil-map guards. 05 refactors are uniformly reasonable; gpt-5.6-luna/gpt-5.5 extracted the most named constants, haiku's `coupon_discounts = {"SAVE5": -5}` negative-value table is a bit too clever.

## 5. Rankings

- **Best correctness:** 8-way tie at 9/9; **gpt-5.4** wins on tiebreak (fastest run, leanest output, 9/9 in both runner harnesses). *Caveat:* one sample at nonzero temperature; a 9/9 vs 8/9 gap is one coin flip.
- **Best correctness-per-dollar (Claude only):** **haiku** — $0.074/pass vs sonnet $0.27, opus $0.30, fable $0.75. *Caveat:* haiku also had the most output tokens and second-slowest wall time in the family, and 7/9 correctness.
- **Fastest (wall/task):** **gpt-5.4** at 3.2s/task. *Caveat:* one API call per task vs agentic tool round-trips; cloud queue variance makes one-shot timings especially noisy (glm one-shot spent 149s on task 09 alone).
- **Most surgical (added lines/task):** **gemma4_cloud-cc** at 14.6 (+131/−82), fable best of Claude at 16.1. *Caveat:* this is churn, not quality — gemma4-cc failed a task, and one-shot models are structurally penalized (whole-file returns get re-diffed but verbose DP rewrites on 09 inflate counts); gpt-5.5's high count is mostly gold-plating, not error.

## 6. Caveats

- **Single sample, sampling temperature:** every pass/fail margin here is one run wide. The 06 result (4/4 Anthropic failures on the same missing guard) looks systematic, but I'd want a re-run before claiming it; the glm-5.2 one-shot-pass/cc-fail split on 06 shows the same weights landing on both sides of the line.
- **Reporting asymmetry:** cost exists only for the four Claude rows; the summed-input column in the raw leaderboard (16.2M) is meaningless as printed — it mixes cached reads, uncached bridge traffic, and 5k one-shot prompts.
- **Whole-file vs surgical:** edit counts and output tokens structurally favor Claude/`-cc` runners; the one-shot adapter's whole-file returns were already flagged as a parsing hazard in this repo's history.
- **Oracle limitations:** (a) **01-lru-cache is saturated** — a one-token fix, 15 identical diffs, zero discrimination; consider retiring or hardening it. (b) **06** is passable by fixes that dodge rather than solve — gemma4's early-return skips locking entirely on self-transfer, and a single global mutex would pass while destroying the parallelism the doc comment promises; the oracle doesn't appear to check disjoint-pair concurrency. (c) **05-god-refactor**'s behavioral oracle would accept a no-op "refactor"; nothing in the visible data grades structure. (d) The shown 06 failure output is *only* the self-transfer test — I can't tell from this data whether the hidden suite also exercises opposing transfers (all failing models did fix that case, so it never fires).
- **07 tells you nothing** — 15 identical fixes; it's a warm-up, not a discriminator. The run's entire signal lives in 06 and 08.
